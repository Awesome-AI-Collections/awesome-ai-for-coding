# LiteEdit

## 一句话总结

一个原生 macOS 轻量代码编辑器，主打亚秒级启动和低资源占用，适合在 AI 编码工作流里承担快速打开、浏览和小改动任务。

## 快速判断

- 如果你想在主 IDE 之外保留一个“秒开、秒关”的副编辑器，它值得优先看
- 如果你需要扩展生态、LSP、终端、Git UI 这类完整 IDE 能力，先看别的方案
- 它更像 AI 编码时代的轻量副编辑器，不是 VS Code / Xcode 的替代品

## 你会怎么用它

- 接进 AI 开发流：让 AI 负责重任务，你用 LiteEdit 快速查看生成文件、配置、小脚本和日志
- 接进日常工作流：作为日常“开 repo 看一眼、改一行、保存退出”的超轻工具
- 最小落地方式：先把它当成主 IDE 旁边的 second editor，用 1 周看自己是否真的更少等待

## 它解决什么问题

- 很多编辑器即使只为改一行配置，也要拉起一整套重量级运行时
- AI 编码时代，开发者经常需要临时打开很多小文件、生成文件和日志，不值得每次都等主 IDE 启动
- 主 IDE 适合深度开发，但不一定适合“快速读一下、改一下、关掉”的微任务

## 适合谁

- 想在主 IDE 之外保留一个零负担副编辑器的开发者
- 重度使用 AI 编码工具，但仍需要极快本地代码浏览与快修的人
- 在 macOS 上追求启动速度和低资源占用的工程师

## 核心能力

- 原生轻量：基于 Swift 和 AppKit 构建，体积很小，启动快
- 快速浏览与编辑：支持文件树、标签页、查找替换、快速打开和行号显示
- 语法高亮：覆盖 20+ 常见语言和配置格式
- 会话保留：可记住文件夹、打开文件、光标位置和窗口状态
- 多光标编辑：支持常见的多点批量修改动作

## 能力边界

- 明显可用：快速查看 repo、阅读配置、改脚本、修 typo、看日志
- 效果一般：需要一定编辑体验但不需要复杂工程能力的中轻量开发任务
- 不要误用：不要把它当成完整 AI IDE，也不要期待插件、LSP、终端或 Git 图形界面

## 集成方式

- 作为单独工具：直接当成主 IDE 旁边的秒开编辑器
- 作为 AI 工作流组件：适合放在“模型生成之后、人类快速复查与小改动”的环节
- 作为团队流程节点：给 Mac 团队补一个低摩擦查看器，减少每次为小改动拉起重量级编辑器

## 上手建议

- 第一步先只拿它处理配置文件、Markdown、JSON、YAML 和日志
- 最值得先试的场景是“AI 生成后的人类快速复查和小改动”
- 如果你已经强依赖扩展、语言服务或终端集成，先轻量试用，不要直接替换主编辑器

## 选型建议

- 如果你的主需求是“极快打开与快修”，适合用它
- 如果你的主需求是完整工程开发，还是更适合看 [Visual Studio Code](visual-studio-code.md) 或 [Cursor](cursor.md)
- 如果你要的是 AI 时代的副编辑器而不是主 IDE，LiteEdit 的价值会比较清楚

## 为什么值得关注

- 它抓住的不是“功能更多”，而是“很多开发任务根本不需要那么重”
- 在 AI 编码流里，快速查看和微调生成文件的频率比以前更高，这类工具的价值会被放大
- README 对“不是什么”写得很明确，不容易误判定位

## 类似项目

- [Visual Studio Code](visual-studio-code.md) - 更适合完整开发和扩展生态；LiteEdit 更适合超轻量快开快关
- [Cursor](cursor.md) - 更偏 AI 原生主力编辑器；LiteEdit 更像极简副编辑器
