# iTerm2

## 一句话总结

macOS 上成熟稳定的高级终端应用，适合作为开发者日常 AI coding 工作台、多会话 shell 和终端协作的基础底座。

## 快速判断

- 如果你在 macOS 上长期用 Claude Code、Codex、各类 shell 和 `tmux` 写代码，它值得收进开发工作台
- 如果你只偶尔开一下终端，不一定能明显感受到它的价值
- 它更像 AI coding 环境的基础设施，而不是 AI 功能工具

## 你会怎么用它

- 接进 AI 开发流：作为本地终端主入口，承载 coding agent、日志窗口、分屏会话和远程 shell
- 接进日常工作流：把不同项目、不同 agent、不同环境固定到标签页和分屏布局中
- 最小落地方式：先把最常用的 AI coding CLI 和远程开发会话切进 iTerm2，观察切换成本是否下降

## 它解决什么问题

- AI coding CLI 和工程终端常常需要长时间驻留，不适合用最基础的系统终端凑合
- 多项目、多会话和远程连接切换频繁时，窗口组织和操作细节很影响效率
- 终端体验不稳定会放大开发与 agent 协作时的摩擦成本

## 适合谁

- 在 macOS 上重度使用终端和 AI coding 工具的开发者
- 需要分屏、标签页和稳定日常体验的终端重度用户
- 把终端看作开发主工作台而不是临时工具的人

## 核心能力

- 成熟工作台能力：适合长期承载多窗口、多标签和多会话终端工作
- 远程与本地并用：方便把 SSH、`tmux` 和本地 coding agent 统一在一个入口里
- 终端细节完善：适合重度 shell 用户长期依赖
- macOS 生态成熟：对苹果平台开发者尤其友好

## 能力边界

- 明显可用：作为 macOS 上 AI coding 和 terminal-first 开发工作流的长期底座
- 效果一般：如果你主要在浏览器里工作，终端只偶尔打开几次，它不是高优先级
- 不要误用：它不是 AI agent，也不负责自动化编排、推理或工具调用

## 集成方式

- 作为单独工具：直接作为默认终端应用使用
- 作为 AI 工作流组件：承载 coding agent、远程开发、日志观察和脚本执行
- 作为团队流程节点：适合团队统一终端工作习惯和会话组织方式

## 上手建议

- 先把 1 个主项目的本地 coding agent 和 1 个远程开发会话迁进去试
- 最值得先试的是分屏 + 标签页组织你的日常 AI coding 工作台
- 如果你当前终端配置已经很复杂，先平移核心工作流，不要一次性折腾全部细节

## 选型建议

- 如果你的主需求是“macOS 上稳定成熟的开发终端底座”，适合用它
- 如果你的主需求是“终端里直接获得 AI 能力”，应该继续找 agent 或 CLI 工具，而不是终端本身
- 如果你是在搭建 AI coding 工作台，它是合理的基础设施项

## 为什么值得关注

- 很多 AI coding 工具的真实体验，最终取决于你每天停留的终端工作台
- 它收录进来不是因为“知名”，而是因为它直接影响开发者与 agent 的日常交互环境
- 对 terminal-first 开发者来说，这类底座工具的复利很高

## 类似项目

- [Ghostty](ghostty.md) - 同样是终端工作台，但更强调现代终端体验和新一代使用感受
- [Visual Studio Code](visual-studio-code.md) - 同样属于开发工作台，但 VS Code 是编辑器中心，iTerm2 是终端中心
