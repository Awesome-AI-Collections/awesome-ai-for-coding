# Visual Studio Code

## 一句话总结

一个开放生态的 AI 代码编辑器与开发平台，在经典 VS Code 编辑体验之上，把 Copilot 本地 Agent、云端 Agent、CLI Agent、扩展和 MCP 能力整合进多 Agent 开发工作流。

## 它解决什么问题

- 很多开发者已经深度绑定 VS Code 生态，但不想为了 AI 工作流彻底迁移到全新编辑器
- 传统 IDE 里的 AI 功能常常割裂在补全、聊天和插件之间，难以形成统一的 agent 体验
- 团队既想保留成熟的扩展生态和熟悉编辑器，又想逐步接入更自主的多 Agent 开发模式

## 适合谁

- 想在熟悉编辑器里逐步接入 AI Agent 的开发者
- 已经使用 VS Code 生态并希望统一编辑、扩展和 Copilot 工作流的团队
- 需要同时兼顾手写代码、扩展生态和多 Agent 协作的人

## 核心能力

- 多种 Agent 形态：覆盖本地 chat / agent、CLI agent、云端 agent 与异步 PR 工作流
- 编辑器核心稳定：延续 VS Code 的编辑、调试、Git、远程开发和语言扩展基础设施
- 扩展能力强：可通过扩展、MCP 和自定义规则把 agent 接入团队自己的开发环境
- 多会话观察：强调在同一视图下管理多个 agent session，而不是只盯着单一聊天窗口

## 典型使用场景

- 在同一个 VS Code 工作区里同时写代码、调用 Copilot agent、查看异步 agent 进展
- 利用扩展和 MCP，把团队已有工具链接进 AI 开发流程
- 对不想脱离主流编辑器生态的团队来说，把 VS Code 作为 AI IDE 基座持续演进

## 为什么值得关注

- VS Code 已经是主流编辑器基座，它一旦把 AI agent 做成主线，就会直接影响大量开发者工作流
- 相比新创 AI IDE，它最大的优势不是新奇，而是编辑器、扩展和远程开发生态的成熟度
- 现在首页已经明确把“open source AI code editor”和“multi-agent development”放到中心叙事，说明路线很清晰

## 类似项目

- [Cursor](cursor.md) - 同样强调 AI IDE 与更自主的 agent 流程，但 Cursor 更像原生 AI-first 产品；VS Code 更依赖成熟编辑器生态和 Copilot 体系做整合。
- [Trae](trae.md) - 同样在往 AI IDE 方向走，但 VS Code 的优势在于开放扩展生态、语言覆盖和团队已有使用基础。
